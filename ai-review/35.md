# Review: ErdŇĎs Problem 35

## 1. Code Reuse

Several definitions in this file duplicate existing infrastructure in the codebase:

**Additive Basis.** `Erdos35.IsAdditiveBasis` is equivalent to `Set.IsAddBasisOfOrder` from `FormalConjecturesForMathlib/Combinatorics/Additive/Basis.lean`. The library version uses `‚ąÄ a, a ‚ąą A ^ n` (pointwise set power), and its equivalence to the `‚ąÉ f : Fin n ‚Üí M, ...` form is proven as `Set.isAddBasisOfOrder_iff` (the `@[to_additive]` of `isMulBasisOfOrder_iff`). The library version is also more general, working over any `CommMonoid`/`AddCommMonoid` rather than just `‚Ąē`.

**Sumset.** `Erdos35.sumset` reinvents Mathlib's pointwise `Set.add` (`A + B` via `open scoped Pointwise`), which is definitionally `{n | ‚ąÉ a ‚ąą A, ‚ąÉ b ‚ąą B, n = a + b}` ‚ÄĒ the same thing.

**Schnirelmann Density.** This one is more nuanced. The codebase provides `Set.partialDensity`, `Set.upperDensity`, and `Set.lowerDensity` in `FormalConjecturesForMathlib/Data/Set/Density.lean`, but these use `limsup`/`liminf` of `|S ‚ą© {x < b}| / |{x < b}|`. Schnirelmann density is the *infimum* (not liminf) of `|A ‚ą© {1,...,N}| / N` for `N ‚Č• 1`, and it starts counting from 1, not 0. So while the existing density infrastructure cannot be directly substituted, it would be worthwhile to add a `schnirelmannDensity` definition to the shared library alongside the existing density concepts, rather than defining it in a local namespace.

**Recommendation:** Replace `IsAdditiveBasis` with `Set.IsAddBasisOfOrder`, replace `sumset` with Mathlib's pointwise `+`, and move `schnirelmannDensity` to the shared density library.

## 6. Correctness

The formalization is **mathematically correct** with one notable redundancy:

### Redundant hypothesis `h0 : (0 : ‚Ąē) ‚ąą B`

The hypothesis `h0 : (0 : ‚Ąē) ‚ąą B` is logically implied by `hB : IsAdditiveBasis B k` together with `hk : k ‚Č• 1`. Here is why: `IsAdditiveBasis B k` says that for every `n : ‚Ąē`, there exists `f : Fin k ‚Üí ‚Ąē` with all `f i ‚ąą B` and `‚ąĎ i, f i = n`. Taking `n = 0`, we need `‚ąĎ i, f i = 0` where each `f i : ‚Ąē`. Since natural numbers are non-negative, this forces every `f i = 0`. Since `k ‚Č• 1`, at least one `f i` exists, so `0 ‚ąą B`. The hypothesis `h0` is therefore redundant and can be safely removed.

The docstring even acknowledges this relationship ("Since $0 \in B$ is assumed separately, 'exactly $k$' is equivalent to 'at most $k$'"), but the implication goes further: `0 ‚ąą B` is not just related ‚ÄĒ it's *entailed*.

### Schnirelmann density definition correctness

The definition `sInf {x : ‚ĄĚ | ‚ąÉ N : ‚Ąē, N ‚Č• 1 ‚ąß x = ((Icc 1 N).filter (¬∑ ‚ąą A)).card / (N : ‚ĄĚ)}` correctly computes `inf_{N ‚Č• 1} |A ‚ą© {1,...,N}| / N`. Note that `Finset.Icc 1 N` in `‚Ąē` gives `{1, 2, ..., N}`, which is the standard convention for Schnirelmann density (excluding 0). This is correct.

One technical subtlety: `sInf` on `‚ĄĚ` for an unbounded-below or empty set returns 0 by convention in Mathlib. The set here is always non-empty (take N = 1) and bounded below by 0, so `sInf` behaves as the genuine infimum. This is fine.

### Main inequality

The statement `schnirelmannDensity (sumset A B) ‚Č• őĪ + őĪ * (1 - őĪ) / (k : ‚ĄĚ)` faithfully captures the problem. Note that when `őĪ = 0` the bound is trivially 0, and when `őĪ = 1` the bound is 1, both of which are consistent. The operator precedence `őĪ * (1 - őĪ) / k` is `(őĪ * (1 - őĪ)) / k` which matches the intended `őĪ(1-őĪ)/k`.
