# Review: ErdŇĎs Problem 656

## 1. Code Reuse

**The local `Nat.upperDensity` definition is redundant and should be replaced.**

- **`FormalConjecturesForMathlib/Data/Set/Density.lean`** provides a generic `Set.upperDensity` (line 51‚Äď53) that computes `atTop.limsup` of `Set.partialDensity`. For `‚Ąē` with the default `A := Set.univ`, this is mathematically equivalent to the ad-hoc `Nat.upperDensity` defined at line 39‚Äď41 of `656.lean`. Both compute `limsup_{N ‚Üí ‚ąě} |A ‚ą© {0,‚Ä¶,N‚ąí1}| / N`; they differ only in surface-level encoding (`Finset.filter`/`Finset.range` vs `Set.interIio`/`ncard`).

- **Problem 109** (`FormalConjectures/ErdosProblems/109.lean`) is the closest sibling problem‚ÄĒthe "ErdŇĎs sumset conjecture"‚ÄĒand it already uses `Set.upperDensity` from the shared library (line 39). Problem 656 should do the same for consistency. In fact, 656 is a strengthening of 109: if `B + B + t ‚äÜ A` with B infinite, one can derive B + C ‚äÜ A with B, C both infinite (take C = B shifted by t). The website explicitly notes the connection to problem 109.

- **Sumset notation**: Problem 109 uses `open Pointwise` and writes `B + C ‚äÜ A` cleanly. Problem 656 instead spells out the sumset condition element-wise (`‚ąÄ b‚āĀ ‚ąą B, ‚ąÄ b‚āā ‚ąą B, ‚ąÉ a ‚ąą A, ‚Ä¶`). While the element-wise form is needed here due to the ‚Ą§-shift, the approach could still be made more idiomatic‚ÄĒsee Readability below.

- **Problems 333, 335, 342, 378** all define their own local `upperDensity` variants. This is a codebase-wide pattern of duplication that should eventually be consolidated around the shared `Set.upperDensity`.

## 4. Readability

The code is reasonably readable, but has some areas for improvement:

- **Verbose sumset encoding**: The condition `‚ąÄ b‚āĀ ‚ąą B, ‚ąÄ b‚āā ‚ąą B, ‚ąÉ a ‚ąą A, (a : ‚Ą§) = ‚ÜĎb‚āĀ + ‚ÜĎb‚āā + t` works but is more complex than necessary. An alternative is to express the containment more directly, e.g., `‚ąÄ b‚āĀ ‚ąą B, ‚ąÄ b‚āā ‚ąą B, (‚ÜĎb‚āĀ + ‚ÜĎb‚āā + t).toNat ‚ąą A ‚ąß 0 ‚Č§ ‚ÜĎb‚āĀ + ‚ÜĎb‚āā + t`, or to cast A to a set of integers and write `(‚ÜĎ) '' A` (the image under the coercion). As it stands, the existential `‚ąÉ a ‚ąą A` is slightly indirect‚ÄĒit asserts the existence of a witness rather than stating membership directly.

- **Explicit `Classical.decPred`**: The density definition on line 41 uses `@Finset.filter _ (¬∑ ‚ąą A) (Classical.decPred (¬∑ ‚ąą A))` instead of just opening `Classical` at the top. Problems 333 uses `open scoped Classical` to avoid this verbosity.

- **Namespace naming**: `Erdos656` is fine and consistent with other problems.

## 6. Correctness

**The formalization is mathematically correct, with one minor stylistic concern.**

**Positive aspects:**

- The `answer(True)` wrapper correctly reflects the affirmative resolution by KMRR24.
- The `@[category research solved, AMS 5 11]` tag is accurate.
- The use of `B : Set ‚Ąē` and `t : ‚Ą§` correctly captures the domain: B lives in ‚Ąē while the shift t is an integer.
- The casting `(a : ‚Ą§) = ‚ÜĎb‚āĀ + ‚ÜĎb‚āā + t` correctly handles the mixed-type arithmetic. The existence of `a ‚ąą A` (a natural number) implicitly enforces that `b‚āĀ + b‚āā + t ‚Č• 0`, which is the right constraint. Since B is infinite and t is fixed, this is satisfiable for any choice of B with sufficiently large minimum element.
- The quantifier structure `‚ąÄ b‚āĀ ‚ąą B, ‚ąÄ b‚āā ‚ąą B` correctly includes the case `b‚āĀ = b‚āā`, matching the standard definition of B + B (the sumset includes `2b` for `b ‚ąą B`).

**Minor concern:**

- The existential formulation `‚ąÉ a ‚ąą A, (a : ‚Ą§) = ‚ÜĎb‚āĀ + ‚ÜĎb‚āā + t` is equivalent to but slightly less direct than asserting membership. It is logically correct: the existence of such an `a` is equivalent to saying the integer `b‚āĀ + b‚āā + t` is non-negative and its natural number representation lies in A. No mathematical content is lost or distorted.

**Verdict: Correct and complete.**
