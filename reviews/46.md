### `answer()` usage

- [ ] **FAIL — wrong answer() case pattern.**

  The **original problem** on [erdosproblems.com/46](https://www.erdosproblems.com/46) is phrased as a yes/no question:

  > "Does every finite colouring of the integers have a monochromatic solution to $1 = \sum 1/n_i$ with $2 \le n_1 < \cdots < n_k$?"

  Per the checklist (§6, "How to decide which case applies"), a "Does..." question is **Case 1** and should use the `answer() ↔ P` pattern. Since it has been solved affirmatively by Croot, it should be:

  ```lean
  theorem erdos_46 : answer(True) ↔
      ∀ (α : Type*) [Finite α] (c : ℕ → α),
        ∃ S : Finset ℕ, S.Nonempty ∧
          (∀ n ∈ S, n ≥ 2) ∧
          (∃ color : α, ∀ n ∈ S, c n = color) ∧
          (∑ n ∈ S, (1 : ℚ) / (n : ℚ)) = 1 := by sorry
  ```

  Currently the theorem uses a direct assertion (Case 3 pattern) with quantifiers as theorem parameters. Two sub-issues:

  1. Missing `answer(True) ↔ ...` wrapper (required for yes/no questions).
  2. Quantifiers `(α : Type*) [Finite α] (c : ℕ → α)` are theorem parameters instead of appearing after `answer()` on the RHS. The checklist common rule states: "Quantifiers go **after** `answer(sorry)`, not as theorem parameters."

  The module docstring rephrases the problem as a direct assertion, but the checklist explicitly says: *"The rule depends on how the **original problem is phrased in English**, not how the Lean code currently looks."*
