### `answer()` usage

The module docstring states: "If T is a tree which is a bipartite graph with k vertices in one class and 2k vertices in the other class then R(T) = 4k − 1." This is phrased as a **direct assertion** (Case 3), not a yes/no question. The conjecture was disproved by Norin, Sun, and Zhao.

Under strict reading of the checklist, **Case 3** applies. For a disproved assertion the expected form is:
```lean
theorem erdos_549 : ¬ (∀ (k : ℕ), k ≥ 1 → ...) := by sorry
```

The file instead uses **Case 1** form: `answer(False) ↔ P`.

- [ ] **Debatable:** The problem is phrased as a direct assertion, suggesting Case 3 (`¬ P` with no `answer()` wrapper). The file uses Case 1 (`answer(False) ↔ P`). Semantically `answer(False) ↔ P` is equivalent to `¬ P`, so the mathematical content is correct. However, the checklist says Case 1 applies only when the problem "asks a yes/no question" ("Does ...", "Is it true that ...", etc.), and the phrasing here is a direct assertion. This is a **minor deviation** — the `answer()` usage is defensible if one reads the conjecture as implicitly asking "Is it true that …?", but does not match the literal phrasing.
- [x] `answer(False)` correctly indicates the conjecture is false (the answer is "no").
- [x] The RHS (`P`) correctly formalizes the original assertion: for every k ≥ 1, every tree T on 3k vertices that is bipartite with classes of size k and 2k satisfies R(T) = 4k − 1.
- [x] Quantifiers go **after** `answer(False)` (inside the RHS of `↔`), not as theorem parameters.
